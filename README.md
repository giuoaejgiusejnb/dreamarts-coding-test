# 最長片道きっぷの旅 - コーディングテスト提出コード

株式会社ドリーム・アーツ 2027年新卒 エンジニア採用コーディングテストの提出物です。
駅と駅を結ぶ鉄道路線網において、同じ点を2回通らずにもっとも長い距離を移動する経路を探索するプログラムです。

## 1. 実行環境

特定のOSやハードウェアへの依存を排除するため、Pythonの標準ライブラリのみを用いて実装しています。

* **OS:** Windows 11 Home 64-bit (クロスプラットフォームを意識し、Linux環境等でも動作するよう標準入出力を実装しています)
* **言語:** Python 3.12
* **外部ライブラリ:** 不要（標準ライブラリ `sys`, `dataclasses` のみ使用）

## 2. 実行手順

お使いの環境（ターミナル・シェル）に合わせて、標準入力にテストデータを流し込んで実行してください。

### Mac / Linux（ターミナル）または Windows（コマンドプロンプト）の場合
```bash
python main.py < input.txt
```

### Windows（PowerShell）の場合
```powershell
Get-Content input.txt | python main.py
```


## 3. 計算量について

### 3.1 時間計算量

`RailwayNetwork` クラスの内部メソッド `_compute_bit_dp` における、以下のループ処理が全体の実行時間を決定します。
```python
for visited_set in range(1 << self.N):
    for current_idx in range(self.N):
        ~中略~
        for railway in self.stations[current_idx].railways:
```

上記のコードにおいて、駅の総数を $N$ とすると、各ループの回数は以下の通りとなります。

- `for visited_set in range(1 << self.N):` は $2^N$ 回
- `for current_idx in range(self.N):` は $N$ 回
- `for railway in self.stations[current_idx].railways:` は 最大 $N$ 回

したがって、全体の時間計算量は **$O(2^N N^2)$** となります。具体的に N の値を入れると以下のようになります。

- $N=18$ のとき： $2^{18} \times 18^2 \approx 8.49 \times 10^7$ 回
- $N=20$ のとき： $2^{20} \times 20^2 \approx 4.19 \times 10^8$ 回

競技プログラミング等で目安とされる実行可能計算回数は $10^8 \sim 10^9$ 回であるため、この実装では $N=18 \sim 20$ ほどが限界となります。

それ以上の $N$ に対しては山登り法や焼きなまし法などの「近似解」を求めるアプローチが必要になりますが、今回は「厳密解」を求める必要があったため、厳密解法のみを採用しました。

### 3.2 空間計算量（メモリ使用量）
本アルゴリズムでは、最長経路の長さを保持する `dp` 配列（浮動小数点数型）と、経路復元用に1つ前の駅を記録する `prev_idxs` 配列（整数型）の2つを使用します。どちらもサイズは $2^N \times N$ であり、全体の空間計算量は **$O(2^N N)$** となります。

$N=20$ の場合を例に挙げると、1つの配列あたり $2^{20} × 20 ≈ 2.10 × 10^{7}$ 個の要素を確保します。2つの配列を合わせると、約 $4.19 × 10^{7}$ 個の要素を保持することになります。

Pythonにおける各データの標準的なメモリサイズ（`float` は 24バイト、`int` は 28バイト）を考慮すると、必要なメモリ容量の目安は以下の通りです。

$(2.10 \times 10^7 \times 24 \text{ バイト}) + (2.10 \times 10^7 \times 28 \text{ バイト}) \approx 1.09 \text{ GB}$

C++などの静的型付け言語（`float` や `int` が4〜8バイト）で実装した場合（約168MB〜336MB）に比べ、Pythonではオブジェクトのオーバーヘッドが大きいため、N=20 の段階で約 1.1 GB のメモリを消費します。
競技プログラミング等の一般的なメモリ制限（512MB 〜 1024MB）を考慮すると、空間計算量の観点からも、Pythonによる本実装の限界目安は **N=18 ~ 19 程度**（ $N=19$ のとき約 530MB）となります。

## 4. 最長経路が複数存在する場合の挙動と拡張性

本実装では、最長経路の長さを持つ経路が複数存在する場合、「見つかった経路のうちの1つのみ」を出力する仕様にしています。もし「すべての最長経路」を求めたい場合の対応策と、それを断念した背景（計算量・メモリのトレードオフ）は以下の通りです。

### 4.1 最終出力部での対応（`_reconstruct_path` メソッド）
メソッド `_reconstruct_path` 内には以下の処理があります。

```python
best_state = best_states[0]  # 今回は一つだけ出力
```

末尾の頂点と訪問状態（ビット集合）の組み合わせにおいて、同じ最長距離を持つパターンが複数ある場合、`best_states` リストにはそのすべてが格納されています。
もしすべての答えを出力したい場合は、この `best_states` のみを処理している部分を、**`best_states` の全要素に対してループ処理を行う**ように変更すれば対応可能です。

### 4.2 DP更新部での課題と断念した理由（`_compute_bit_dp` メソッド）
しかし、完全にすべての最長経路を網羅するためには、DPの更新部（`_compute_bit_dp` メソッド）にも変更が必要になります。現状は以下のようになっています。

```python
if dp[next_visited_set][next_idx] < new_length:
    # dpを更新
    dp[next_visited_set][next_idx] = new_length
    # 経路復元用に前の駅のインデックスを記録
    prev_idxs[next_visited_set][next_idx] = current_idx
```

同じ最長距離となる「別の経路（前の駅）」もすべて記録する場合、条件式を `<` から `<=` に変更した上で、`prev_idxs[next_visited_set][next_idx]` を単一の数値ではなくインデックスのリストとして保持しなければなりません。

この変更を行うと、以下の問題が発生します。
1. **空間計算量の増加**：各状態で保持するデータのサイズがリスト化によって膨らみ、ただでさえ大きい Python のメモリ消費（空間計算量）をさらに圧迫する。
2. **時間計算量の増加**：経路復元（バックトラック）の際に探索ツリーが分岐するため、最長経路の数に応じて処理時間が大幅に増加する。

今回は「厳密解を求めること」が最優先であり、かつ 3章 で述べた通り $N=18 \sim 20$ の段階で計算量・メモリ使用量が限界に達するため、リソースの肥大化を避ける目的から「1つのみを出力する仕様」に留め、すべての経路網羅は断念しました。


## 5. 設計思想とコードの可読性について

一般的な競技プログラミング（制限時間が短い環境）においては、実装スピードが最優先されるため、変数名を極端に短くしたり、クラス化を行わずグローバル変数や関数のみで記述したりすることが多く、コードの可読性や再利用性が犠牲になりがちです。

しかし、今回の開発においては十分な制作・推敲期間が確保されていたため、以下の点に注力して実装を行いました。

- **明瞭な命名規則（Naming Convention）**
  - 変数名やメソッド名（`visited_set`、`current_idx`、`_compute_bit_dp` など）に役割がひと目でわかる具体的な名称を採用し、コード自体の自己文書化を図りました。
- **オブジェクト指向に基づくクラス化**
  - 役割ごとにデータと処理を `Railway`、`Station`、`RailwayNetwork` という適切なクラスにカプセル化しました。

これにより、アルゴリズムの挙動が他者から見ても理解しやすくなり、将来的な仕様変更や機能拡張（例：経路の重み付けルールの変更など）にも柔軟に対応できる、保守性の高いプログラムとして仕上げることができました。
